# tools/

Scripts for regenerating the generated inputs that `libadwaita/src/*.hg` files rely on: the `libadwaita_*.defs` files, and the doc-comment database gmmproc pulls Doxygen comments from.

## The `libadwaita_*.defs` files

`libadwaita/src/` contains five `libadwaita_*.defs` files that gmmproc reads (via `_DEFS(libadwaitamm,libadwaita)` in the `.hg` files) to know what the C API looks like.
They are not all produced the same way:

| File                      | How it’s produced                                                   |
| ------------------------- | ------------------------------------------------------------------- |
| `libadwaita.defs`         | **Hand-written.** Just `(include ...)`s the other four.             |
| `libadwaita_vfuncs.defs`  | Generated by `generate_vfuncs_defs.py` (see below).                 |
| `libadwaita_enums.defs`   | Generated by `generate_defs_and_docs.sh` (via glibmm’s `enum.pl`).  |
| `libadwaita_methods.defs` | Generated by `generate_defs_and_docs.sh` (via glibmm’s `h2def.py`). |
| `libadwaita_signals.defs` | Generated by `generate_defs_libadwaita` (see below).                |

`libadwaita.defs` is the only one left to hand-maintain.

Regenerate after upgrading the libadwaita version this project binds against (new enums, methods, properties, signals, or vfuncs to wrap), or after adding a new class to `libadwaita/src/*.hg` (also update `generate_defs_libadwaita.cc`’s type list first, see below).

### `enums.defs`, `methods.defs` (and `docs.xml`, see “Documentation” below)

```sh
tools/generate_defs_and_docs.sh
```

This assumes the classic \*mm jhbuild layout: a sibling `glibmm` checkout (for `glibmm/tools/defs_gen/{h2def.py,docextract_to_xml.py}` and
`glibmm/tools/enum.pl`) and the _system-installed_ libadwaita headers under `/usr/include/libadwaita-1/`.
Point `GMMPROC_GEN_SOURCE_DIR`/`GMMPROC_GEN_BUILD_DIR` at your jhbuild `checkoutroot`/`buildroot` if you’re not using the default layout — see the comment block at the top of the script.

### signals.defs (properties and signals)

`generate_defs_libadwaita` (built from `tools/extra_defs_gen/generate_defs_libadwaita.cc`) calls glibmm’s `get_defs()` on every `ADW_TYPE_*` GObject type wrapped by this project and prints its properties/signals in `.defs` form.

With a configured Meson build directory in `maintainer_mode`, this is one command:

```sh
ninja -C <builddir> gen-signals
```

That runs `tools/generate_signals.sh`, which just pipes the executable’s output to `libadwaita/src/libadwaita_signals.defs`.
Without Meson (e.g. jhbuild), `generate_defs_and_docs.sh` calls the same executable itself.

**Important:** `generate_defs_libadwaita.cc` lists every wrapped type by hand, alphabetically by type name (not grouped by `.hg` file).
When you add a new `.hg` file (or a new class within an existing one), add its `ADW_TYPE_*` there too — otherwise its properties and signals silently never make it into `libadwaita_signals.defs`.

### `vfuncs.defs` (virtual functions)

`tools/generate_vfuncs_defs.py` scans installed libadwaita headers for `AdwFooClass` struct vfuncs (function-pointer members up to a `/*< private >*/` marker) and emits a `(define-vfunc ...)` for every one it finds, dropping the leading `self` parameter (implicit via `of-object`, like `define-method`) and converting common C/GLib types to gmmproc `.defs` spelling (e.g. `const char *` → `const-gchar*`) via a small alias table in the script – extend it there if a type comes out wrong.
This is a full, unconditional regeneration, like the other three generated files: it doesn’t know or care which vfuncs already have a `_WRAP_VFUNC(...)` in some `.hg` file, it just lists everything found in the given headers.

```sh
# Using installed libadwaita-1 headers (default search path):
ninja -C <builddir> gen-vfuncs
# or directly, e.g. against a local header checkout:
tools/generate_vfuncs_defs.py path/to/headers/*.h
# preview without writing:
tools/generate_vfuncs_defs.py --dry-run
```

A `define-vfunc` entry existing doesn’t wrap it for you – most vfuncs still need a `_WRAP_VFUNC(...)` call written by hand in the relevant `.hg` file (grep `libadwaita/src/*.hg` for examples); the generated `.defs` entry is what `_WRAP_VFUNC` (and any default-handler logic gmmproc derives from it) looks up by name.

## Documentation generation

There are two separate, easily-confused “documentation” pipelines here:

1. **Doc-comment extraction (`libadwaita_docs.xml`)**: pulls the gtk-doc/gi-docgen `/**  */` comments out of the libadwaita C sources into an XML database.
   gmmproc’s WrapParser looks entries up in it by C function/property/signal name to auto-fill a Doxygen comment on the generated C++ `_WRAP_METHOD`/`_WRAP_PROPERTY`/`_WRAP_SIGNAL` calls that don’t already have a hand-written one (see the hand-written Doxygen blocks already present in some `.hg` files for the ones that need something more specific than the auto-pulled comment).
   Regenerated together with `enums.defs`/`methods.defs` by `tools/generate_defs_and_docs.sh` — see above; it’s not a `.defs` file, but it’s produced by the same script and consumed by gmmproc the same way (`libadwaita_docs.xml` is listed alongside the `.defs` files in `libadwaita/libadwaitamm/meson.build`’s `defs_basefiles`).

2. **Reference manual (`doc/reference/`)**: runs Doxygen over libadwaitamm’s _own_ generated and hand-written C++ headers to build the HTML/Devhelp reference manual for this project.
   Driven by `doc/reference/meson.build` and `doc/reference/Doxyfile.in`; it builds automatically as part of `ninja -C <builddir>` whenever Meson’s
   `build_documentation` option is enabled (it’s the `html_and_tag` custom_target, `build_by_default: build_documentation`).
   This consumes the Doxygen comments that step 1 helped fill in — it doesn’t read `libadwaita_docs.xml` directly.

In short: step 1 is “how do C doc comments get into the generated C++ headers,” step 2 is “how do those C++ headers become an HTML reference manual.” Regenerating step 1’s output only matters if you’ve changed doc comments in the libadwaita C sources (e.g. after a libadwaita upgrade); step 2 rebuilds automatically from whatever’s currently in the `.hg`/`.h` files.
